## SCSS+BEM 경험자가 Tailwind를 써본 후기

몇 년간 SCSS+BEM에 익숙해진 상황에서 Tailwind CSS를 접했습니다.

처음 Tailwind 봤을 때 "이게 뭐야, inline style이랑 뭐가 달라?" 생각도 했습니다.

비교해보자면...

```tsx
// Tailwind
<div className="flex items-center gap-2 text-blue-500">

// Inline style
<div style={{ display: 'flex', alignItems: 'center', gap: '8px', color: '#3b82f6' }}>
```

두 예시문 모두 **어떻게 보이는가**에 집중하고 있습니다.

저는 `.card__title--highlighted` 같은 의미 있는 클래스명에 익숙했습니다.

"**이 요소가 무엇인가**"에 집중하는 방식이요.

하지만 Tailwindcss 쓰면서 달랐습니다.

빠릅니다. 정말로 매우 빨랐습니다.

그래서 생긴 의문 중 하나는... 이거 이대로 이렇게 사용해도 괜찮은 걸까?

문득 드는 생각은 OOP에서 **상속 vs 컴포지션** 논쟁이 있는 것처럼, 계층이 깊어지면 복잡해지니까 조합(Composition)을 쓰자는 흐름.

이것을 CSS에도 다음과 같이 비교해볼 수 있을 것 같습니다.

|           | OOP         | CSS                           |
| --------- | ----------- | ----------------------------- |
| 컴포지션  | 부품 조합   | Tailwind (유틸리티 조합)      |
| 상속/구조 | 계층 + 규칙 | SCSS+BEM (중첩 + 네이밍 규칙) |

<!--truncate-->

## 두 프로젝트 이야기

### 빠른 일정 프로젝트 — Tailwind

디자인 시안 없이 2개월 안에 끝내야 하는 프로젝트가 있었습니다.

일정이 최우선.

솔직히? Tailwind 아니었으면 못 맞췄을 수도 있었습니다.

결과물은 이랬습니다.

```tsx
className = "mt-[26px] text-[15px] font-[400] text-[#222225]";
```

arbitrary value 형태의 사용이 매우 많았습니다.

`[26px]`, `[#222225]`. 디자인 토큰 없이 숫자를 때려박았습니다.

하나의 클래스면 괜찮지만, 여러 태그가 중첩되면?

읽기 어렵습니다. 나중에 보면.

**하지만 일정 안에 끝냈습니다.**

그게 제일 중요했지요.

여기서 Tailwind의 빛을 봤습니다.

### 장기 운영 서비스 — SCSS + BEM

반대로 2년 넘게 SCSS+BEM으로 유지보수해온 서비스도 있습니다.

```scss
// _button.scss
.btn {
  &--primary {
    background-color: $c-primary;
    color: $c-white;
  }
  &--secondary {
    background-color: transparent;
    color: $c-primary;
  }
  &--small {
    height: 3.4rem;
    font-size: $f-footnote;
  }
  &--medium {
    height: 3.8rem;
    font-size: $f-contents;
  }
}
```

`.btn--primary`, `.btn--small`.

클래스만 봐도 뭔지 안다.

2년 넘게 여러 개발자가 유지보수해도 구조가 안 깨졌다.

### 비교

| 프로젝트           | 스타일링   | 기간  | 목적 |
| ------------------ | ---------- | ----- | ---- |
| 빠른 일정 프로젝트 | Tailwind   | 2개월 | 오픈 |
| 장기 운영 서비스   | SCSS + BEM | 2년+  | 운영 |

## Tailwind가 좋았던 점

### 1. 네이밍 고민이 사라졌다

BEM 쓸 때 클래스명의 정의는 생각보다 많은 비용이 들어갑니다.

`card-item`? `card__item`?

`--active`? `--is-active`?

Tailwind 쓰면 이 고민이 없습니다.

그냥 `flex items-center gap-2` 치면 끝.

### 2. 팀 온보딩이 빨랐다

새로운 팀원이 왔을 때.

BEM이면 "우리 팀 규칙은 이래요" 설명해야 한다.

Tailwind는 공식 문서 링크 주면 끝.

### 3. inline style과는 다르다

"inline style이랑 뭐가 달라?"

Tailwind 창시자인 Adam Wathan은 이렇게 말했어요.

> "Utilities differ from inline styles because they enforce constraints."
> (유틸리티는 제약을 강제하기 때문에 인라인 스타일과 다르다)

`text-sm`이냐 `text-xs`냐. 제한된 선택지에서 골라요.

`font-size: 13px` vs `14px` 같은 무한한 선택과는 달라요.

하지만 `sm`, `xs`도 결국 표현에 집중하는 값이라는 생각입니다.

저는 여전히 "무엇을 의미하는가"에 무게를 두고 싶습니다.

`.text--title`, `.text--caption` 처럼요.

## 편하지만 불편했던 점도 있다

**디자인 시스템 없이 시작하면 arbitrary value 지옥.**

### 1. arbitrary value가 쌓인다

```tsx
className = "text-[#222225]"; // 30+ 곳
className = "bg-[#f5f5f5]"; // 20+ 곳
className = "mt-[26px]";
className = "text-[15px]";
```

나중에 색상 바꾸려면?

전체 파일 검색해서 하나씩 고쳐야 합니다.

### 2. 현업의 컴포넌트는 단순하지 않다

일부는 "의미는 컴포넌트에 위임하면 돼요."

맞는 말이지만, 현업에서 일하다 보면 불가피하게 HTML과 뒤엉킨 계층 구조가 생기기 마련입니다.

```tsx
<div className="relative flex items-center justify-between p-4 rounded-lg bg-white shadow-md">
  <div className="flex items-center gap-3">
    <img className="w-12 h-12 rounded-full object-cover" />
    <div className="flex flex-col">
      <span className="text-sm font-medium text-gray-900">...</span>
      <span className="text-xs text-gray-500">...</span>
    </div>
  </div>
  <button className="px-3 py-1.5 text-sm font-medium text-white bg-blue-500 rounded-md hover:bg-blue-600">...</button>
</div>
```

이 정도면 그래도 양호하다고 생각합니다. 하지만 클래스명이 길어지면 **읽기가 어렵다.**

### 3. CSS 선택자를 포기해야 한다

아래 CSS의 강력한 선택자들은 매우 유용하게 종종 사용하곤 했습니다.

```css
li + li {
  margin-top: 8px;
}
:nth-child(3n + 1) {
  grid-column: 1;
}
input:focus ~ .label {
  transform: translateY(-20px);
}
```

`+`, `~`, `:nth-child`, `:has`.

하지만 Tailwind에서는?

```html
<li class="[&+&]:border-t [&+&]:border-gray-200"></li>
```

읽기도 어렵고, CSS의 선택자들을 무용지물로 만드는 행위입니다.

물론 우회 방법이 있긴 합니다만, 근데 그렇게까지 해가면서 Tailwind를 쓰는 이유인가 싶습니다.

### 4. 장기 운영하면 결국 추상화한다

Vercel, GitHub 같은 장기 운영 프로젝트들.

결국 `<Button variant="primary" />` 형태로 추상화합니다.

결국 BEM이 하려던 걸 컴포넌트 레벨에서 다시 하는 거라는 생각입니다.

그리고 컴포넌트 내부엔 여전히 계층 구조가 있습니다.

```tsx
// Button 컴포넌트 내부
<button className="flex items-center gap-2 px-4 py-2 ...">
  <span className="text-sm font-medium ...">{children}</span>
  {icon && <Icon className="w-4 h-4 ..." />}
</button>
```

결국 "의미"를 어딘가에 담아야 합니다.

클래스명이든, 컴포넌트 이름이든, prop 이름이든.

## Tailwind가 인기 있는 이유

**시장 환경과 맞아떨어졌다.**

### 1. "일단 빨리" 문화

| 환경      | 특징                                 |
| --------- | ------------------------------------ |
| 스타트업  | 낮은 러닝커브와 빠른 출시 > 유지보수 |
| 에이전시  | 납품 후 끝                           |
| 빠른 검증 | 일단 만들고, 리팩토링은 나중         |

### 2. CSS 진입장벽을 낮춤

- BEM 규칙 학습 불필요
- 네이밍 고민 제거
- CSS 잘 몰라도 클래스 조합으로 스타일링 가능

주니어나 백엔드 출신 프론트엔드에게 매력적이라고 생각합니다.

## 그래서 뭘 써야 하나?

**"어떤 도구가 더 좋은가?"**

질문이 틀렸습니다.

그 전에 물어야 할 게 있습니다.

**읽기 쉬운가?**

> "시맨틱 네이밍을 포기하면, 팀원들이 코드를 논의하고 찾는 데 필요한 공통 언어를 잃는다."
> — [Aleksandr Hovhannisyan](https://www.aleksandrhovhannisyan.com/blog/why-i-dont-like-tailwind-css/)

> "코드를 작성하지 않은 팀원이 코드를 물려받을 때, 시맨틱 이름 없이는 상당한 인지적 부담이 생긴다."
> — [thoughtbot](https://thoughtbot.com/blog/tailwind-versus-bem)

읽기 어려우면 수정도, 확장도, 재사용도 기대하기 어렵습니다.

```
BEM: "이 요소가 무엇인가" (Semantic)
Tailwind: "이 요소가 어떻게 보이는가" (Visual)
```

둘 다 맞는 접근.

상황이 다를 뿐.

"둘 다 조화하면 되지 않나?"

이 조화하는 과정에서 규칙 만들고 합의하는 비용이 생기기 마련입니다.

그래서 자연스럽게 극단적 선택이 나오기 마련입니다.

풀 Tailwind면 규칙 자체가 필요 없고, 풀 BEM이면 규칙이 명확하니까.

| 상황                        | 권장       |
| --------------------------- | ---------- |
| 빠른 개발, 프로토타입       | Tailwind   |
| 장기 운영, 디자인 시스템    | SCSS + BEM |
| 팀 온보딩 중요              | Tailwind   |
| 코드 리뷰/커뮤니케이션 중요 | BEM        |

---

## 나의 선택

**기본기는 SCSS+BEM. Tailwind는 단기전 무기.**

장기 운영? 구조와 네이밍이 있는 쪽이 유리하다.

빠른 출시? Tailwind로 속도 낸다. 실제로 그렇게 했다.

중요한 건 "**왜 이걸 선택했는가**" 설명할 수 있는 것.

한 가지 더.

결국 도구다.

원칙 따지다가 제품 못 만들면 본말전도.
