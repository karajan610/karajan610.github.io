당신은 Tailwind CSS를 왜 사용하나요?

2026년 1월 5일 · 약 6분

SCSS+BEM 경험자가 Tailwind를 써본 후기​

몇 년간 SCSS+BEM에 익숙해진 상황에서 Tailwind CSS를 접했습니다.

처음 Tailwind 봤을 때 "이게 뭐야, inline style이랑 뭐가 달라?" 생각했어요.

// Tailwind
<div className="flex items-center gap-2 text-blue-500">

// Inline style
<div style={{ display: 'flex', alignItems: 'center', gap: '8px', color: '#3b82f6' }}>

둘 다 "어떻게 보이는가"를 말하고 있어요.

.card__title--highlighted 같은 의미 있는 클래스명에 익숙했거든요.

"이 요소가 무엇인가"를 말하는 방식이요.

근데 실제로 써보니까 달랐습니다.

빠르더라고요. 진짜로.

그래서 의문이 생겼어요.

이거 괜찮은 건가?

OOP에서 상속 vs 컴포지션 논쟁이 있는것처럼, 계층이 깊어지면 복잡해지니까 조합(Composition)을 쓰자는 흐름.

이것을 CSS에도 다음과 같이 비교해볼수있을것 같습니다.

OOPCSS
컴포지션부품 조합Tailwind (유틸리티 조합)
상속/구조계층 + 규칙SCSS+BEM (중첩 + 네이밍 규칙)

두 프로젝트 이야기​

각각 맞는 선택이었어요.

빠른 일정 프로젝트 — Tailwind​

디자인 시안 없이 2개월 안에 끝내야 하는 프로젝트가 있었습니다.

일정이 최우선.

솔직히? Tailwind 아니었으면 못 맞췄을 수도 있어요.

결과물은 이랬습니다.

className = "mt-[26px] text-[15px] font-[400] text-[#222225]";

arbitrary value 범벅이에요.

[26px], [#222225]. 디자인 토큰 없이 숫자를 때려박았죠.

하나의 클래스면 괜찮지만, 여러 태그가 중첩되면?

읽기 어려워요. 나중에 보면.

하지만 일정 안에 끝냈습니다.

그게 제일 중요했어요.

여기서 Tailwind의 빛을 봤습니다.

장기 운영 서비스 — SCSS + BEM​

반대로 2년 넘게 SCSS+BEM으로 유지보수해온 서비스도 있어요.

// _button.scss
.btn {
  &--primary {
    background-color: $c-primary;
    color: $c-white;
  }
  &--secondary {
    background-color: transparent;
    color: $c-primary;
  }
  &--small {
    height: 3.4rem;
    font-size: $f-footnote;
  }
  &--medium {
    height: 3.8rem;
    font-size: $f-contents;
  }
}

.btn--primary, .btn--small.

클래스만 봐도 뭔지 안다.

2년 넘게 여러 개발자가 유지보수해도 구조가 안 깨졌다.

비교​

프로젝트스타일링기간목적
빠른 일정 프로젝트Tailwind2개월오픈
장기 운영 서비스SCSS + BEM2년+운영

Tailwind가 좋았던 점​

1. 네이밍 고민이 사라졌다​

BEM 쓸 때 클래스명 짓느라 얼마나 시간 썼는지 몰라요.

card-item? card__item? cardItem?

--active? --is-active? --selected?

Tailwind 쓰면 이 고민이 없어요.

그냥 flex items-center gap-2 치면 끝.

2. 팀 온보딩이 빨랐다​

새로운 팀원이 왔을 때.

BEM이면 "우리 팀 규칙은 이래요" 설명해야 한다.

Tailwind는 공식 문서 링크 주면 끝.

3. inline style과는 다르다​

"inline style이랑 뭐가 달라?"

처음엔 저도 그렇게 생각했어요.

Adam Wathan은 이렇게 말했어요.

"Utilities differ from inline styles because they enforce constraints." (유틸리티는 제약을 강제하기 때문에 인라인 스타일과 다르다)

text-sm이냐 text-xs냐. 제한된 선택지에서 골라요.

font-size: 13px vs 14px 같은 무한한 선택과는 달라요.

이게 일관성을 만들어요.

4~5년 전 에피소드가 있어요.

주간 개발회의에서 나온 논쟁. "text-color-red와 text-color-primary, 차이가 뭐냐?"

아무도 대답 못 했다. 나 포함.

지금 생각해보면 red는 값이고, primary는 의미다.

primary가 빨간색에서 파란색으로 바뀌어도 클래스명은 그대로.

Tailwind의 constraints도 같은 맥락. 제한된 선택지가 일관성을 만들어요.

물론 이건 Tailwind 옹호 입장이에요. 다른 생각도 있어요.

편하지만 불편했던 점도 있다​

디자인 시스템 없이 시작하면 arbitrary value 지옥.

1. arbitrary value가 쌓인다​

className = "text-[#222225]"; // 30+ 곳
className = "bg-[#f5f5f5]"; // 20+ 곳
className = "mt-[26px]";
className = "text-[15px]";

나중에 색상 바꾸려면?

전체 파일 검색해서 하나씩 고쳐야 해요.

2. 현업의 컴포넌트는 단순하지 않다​

"의미는 컴포넌트에 위임하면 돼요."

맞는 말이에요. 이상적으로는.

근데 작업하다 보면 계층 구조가 생겨요.

HTML 태그도 여러 개 섞여요.

<div className="relative flex items-center justify-between p-4 rounded-lg bg-white shadow-md">
  <div className="flex items-center gap-3">
    <img className="w-12 h-12 rounded-full object-cover" />
    <div className="flex flex-col">
      <span className="text-sm font-medium text-gray-900">...</span>
      <span className="text-xs text-gray-500">...</span>
    </div>
  </div>
  <button className="px-3 py-1.5 text-sm font-medium text-white bg-blue-500 rounded-md hover:bg-blue-600">...</button>
</div>

이 정도면 양반이다.

클래스명이 길어지면 읽기가 어렵다.

코드 리뷰할 때 힘들어요.

3. CSS 선택자를 포기해야 한다​

CSS의 강력한 선택자들.

li + li {
  margin-top: 8px;
}
:nth-child(3n + 1) {
  grid-column: 1;
}
input:focus ~ .label {
  transform: translateY(-20px);
}

+, ~, :nth-child, :has.

Tailwind에서는?

<li class="[&+&]:border-t [&+&]:border-gray-200"></li>

가독성이 끔찍하다.

우회 방법이 있긴 해요.

근데 그게 Tailwind를 쓰는 이유인가?

4. 장기 운영하면 결국 추상화한다​

Vercel, GitHub 같은 장기 운영 프로젝트들.

결국 <Button variant="primary" /> 형태로 추상화해요.

BEM이 하려던 걸 컴포넌트 레벨에서 다시 하는 거예요.

그리고 컴포넌트 내부엔 여전히 계층 구조가 있어요.

// Button 컴포넌트 내부
<button className="flex items-center gap-2 px-4 py-2 ...">
  <span className="text-sm font-medium ...">{children}</span>
  {icon && <Icon className="w-4 h-4 ..." />}
</button>

결국 "의미"를 어딘가에 담아야 해요.

클래스명이든, 컴포넌트 이름이든, prop 이름이든.

Tailwind가 인기 있는 이유​

시장 환경과 맞아떨어졌다.

1. "일단 빨리" 문화​

환경특징
스타트업빠른 출시 > 유지보수
에이전시납품 후 끝
빠른 검증일단 만들고, 리팩토링은 나중

나도 빠른 일정 프로젝트에서 이 혜택 봤어요.

2. CSS 진입장벽을 낮춤​

  • BEM 규칙 학습 불필요
  • 네이밍 고민 제거
  • CSS 잘 몰라도 클래스 조합으로 스타일링 가능

주니어나 백엔드 출신 프론트엔드에게 매력적이에요.

3. 생태계 효과​

Vercel + Next.js + Tailwind 세트.

Tailwind 기반 생태계가 커지면서 선택지가 늘었어요.

쓰는 사람이 많으니까 더 쓰게 되는 흐름.

그래서 뭘 써야 하나?​

"어떤 도구가 더 좋은가?"

질문이 틀렸다.

그 전에 물어야 할 게 있어요.

읽기 쉬운가?

"시맨틱 네이밍을 포기하면, 팀원들이 코드를 논의하고 찾는 데 필요한 공통 언어를 잃는다." — Aleksandr Hovhannisyan

"코드를 작성하지 않은 팀원이 코드를 물려받을 때, 시맨틱 이름 없이는 상당한 인지적 부담이 생긴다." — thoughtbot

읽기 어려우면 수정도, 확장도, 재사용도 기대하기 어려워요.

BEM: "이 요소가 무엇인가" (Semantic)
Tailwind: "이 요소가 어떻게 보이는가" (Visual)

둘 다 맞는 접근.

상황이 다를 뿐.

"둘 다 조화하면 되지 않나?"

해봤다. 안 됐다.

"이건 컴포넌트로, 저건 유틸리티로" — 기준이 사람마다 달랐어요.

규칙 만들고 합의하는 비용이 생겨요.

그래서 극단적 선택이 나와요.

풀 Tailwind면 규칙 자체가 필요 없고, 풀 BEM이면 규칙이 명확하니까.

상황권장
빠른 개발, 프로토타입Tailwind
장기 운영, 디자인 시스템SCSS + BEM
팀 온보딩 중요Tailwind
코드 리뷰/커뮤니케이션 중요BEM

나의 선택​

기본기는 SCSS+BEM. Tailwind는 단기전 무기.

장기 운영? 구조와 네이밍이 있는 쪽이 유리하다.

빠른 출시? Tailwind로 속도 낸다. 실제로 그렇게 했다.

중요한 건 "왜 이걸 선택했는가" 설명할 수 있는 것.

한 가지 더.

결국 도구다.

원칙 따지다가 제품 못 만들면 본말전도.

당신은 Tailwind CSS를 왜 사용하나요?
