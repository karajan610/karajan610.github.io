전역상태의 관리가 필요할까?

대표적인 상태관리 라이브러리들​

  • Redux

    • 리덕스 초기에 많은 셋팅을 해야한다.
    • 초기에 허들이 있다.
    • Flux패턴 사용..
    • Flux패턴에 익숙해져있다면 Zustand 그렇지 않으면 Jotai 추천..

    둘의 차이는 Jotial는 bottom-up방식... Zustand는 top-down 방식으로 상태 관리하는것...

  • Mobx

  • Zuntand

  • Jotai

  • Context Api

  • 그외...

하지만 근본적으로 이렇게 전역상태를 관리하면 복잡해지는 경우가 다시 생긴다. 그래서 다시 생기는 고민은 전역상태를 해야할까...?

전역상테 중에서도 서버에서 값을 가지고 와서 그걸 조합해서 사용하는 경우가 있는데, 그러면 서버 값을 갖고와서 캐싱을 하는건 어떨까하는것이 나왔고, 그것이 지금 라이브러리로 React-Query, SWR이 된다.

이로써 React-query를 통해서 전역상태관리하던 부분을 많이 대체를 하기 때문에, 사실은 되게 간단하게 쓰이는 부분을 쓸 때 이제 Jotai를 많이 조합하여 쓰는 편이다.

실제 React-query와 Jotai를 조합하여 사용하는 경우도 많다.

  • React-query + Jotai
  • React-query + Zuntand
  • SWR + Jotai
  • SWR + Zuntand

결국 전역상태라는것은 회사마다 사정이 다르다. 사업이 오래된경우 Mobx나 Redux를 사용할 수 있다.

상태라는 것이 많이 존재한다. 예를 들면 UI를 다룰 때 버튼 하나 눌러도 그게 상태가 될 수 있다.

여기서 전역상태관리라는것은 로그인이 가장 대표적이된다. 즉 로그인 상태에 따라서 어드민 같은경우, 판매자만 들어가야 하는 화면이 될 수 있고, 또다른 경우 마스터의경우 모두다 접근이 허용되기도하고, 또, 개발자면 개발자만 보일 수 있는 메뉴나 페이지가 있을 수 있는것이고... 이런식으로 좀 나눠야 할 필요가 있을 때, 이러한 것은 전역으로 관리를 해야해서 이러한 상태관리를 많이 하게된다.
